# Generated Skill Template

Template and assembly instructions for the generated skill's SKILL.md and directory structure.

## Output Directory Structure

```
{output-dir}/
├── SKILL.md                        # Progressive disclosure index
├── openapi.yaml                    # OpenAPI 3.1 specification
└── references/
    ├── quick-start.md              # Auth, headers, errors, pagination
    ├── {controller-name}-api.md    # Per-controller documentation
    └── shared-models.md            # All shared DTOs, entities, enums
```

## SKILL.md Frontmatter

```yaml
---
name: "{project-name}-api-blueprint"
description: "This skill should be used when the user needs to {trigger phrase 1}, {trigger phrase 2}, {trigger phrase 3}. It provides complete endpoint documentation, request/response models, and example payloads for all {N} endpoints across {M} controllers in the {API Display Name} backend."
version: "1.0.0"
generated_by: "api-blueprint"
generated_at: "{ISO 8601 timestamp}"
source_repo: "{repository name or path if available}"
---
```

The `name` field must be `{project-name}-api-blueprint` where `{project-name}` is the kebab-cased working directory name (e.g., a project in `/home/user/projects/core` → `core-api-blueprint`, a project in `/home/user/my-service` → `my-service-api-blueprint`).

### Trigger Phrase Generation

Generate trigger phrases based on the actual controllers and API capabilities. Combine generic API interaction phrases with domain-specific ones.

**Always include these generic phrases:**
- "interact with the {API name} API"
- "make a request to the {API name} backend"
- "call the {API name} endpoints"
- "create models for the {API name} API"

**Generate domain-specific phrases from controllers:**

For each controller, generate 1-2 trigger phrases based on its resource type:
- UserController → "manage users via API", "authenticate with the backend"
- CharacterSheetController → "create or update character sheets", "fetch character data"
- CampaignController → "manage campaigns via API"
- WeaponController → "query available weapons"

Include the most important 6-10 trigger phrases in the description. Prioritize the ones most likely to match what a client developer would ask.

## Progressive Disclosure Rules

The generated SKILL.md MUST follow strict progressive disclosure:

### Always Loaded (SKILL.md)
- API name, base URL, auth summary
- Endpoint index table (method, path, one-line description) — NO request/response details
- Pointers to reference files
- Total under 3000 words regardless of API size

### Loaded on Demand (references/)
- `quick-start.md` — loaded when setting up API integration for the first time
- `{controller}-api.md` — loaded when working with that specific controller's endpoints
- `shared-models.md` — loaded when building model/type definitions or when a model spans controllers
- `openapi.yaml` — loaded when generating client code or importing into API tools

### What NEVER Goes in SKILL.md
- Request/response body examples
- Full model field tables
- Enum value lists
- Curl examples
- Error response details

These all belong in the per-controller reference files.

## Size Guidelines

| File | Target Size | Maximum |
|------|-------------|---------|
| SKILL.md | 500-1500 words | 3000 words |
| quick-start.md | 500-1000 words | 2000 words |
| Per-controller reference | 500-3000 words | No limit |
| shared-models.md | 500-2000 words | No limit |

If the SKILL.md endpoint index would exceed 3000 words (very large API with 50+ endpoints), group controllers by domain area and collapse the table:

```markdown
### User Management (3 controllers, 14 endpoints)
- `references/auth-api.md` — Login, registration, logout
- `references/users-api.md` — User CRUD, profile management
- `references/admin-api.md` — Admin user management
```

## Validation Checklist

Before finalizing the generated skill, verify:

- [ ] SKILL.md has valid YAML frontmatter with name, description, and trigger phrases
- [ ] Every file listed in the Reference Files table exists
- [ ] Every controller from the discovery manifest has a reference file
- [ ] Total endpoint count in SKILL.md matches the sum across all reference files
- [ ] All enum types have every value listed (cross-reference with source)
- [ ] Example JSON in reference files is valid JSON (no trailing commas, correct types)
- [ ] openapi.yaml is valid OpenAPI 3.1
- [ ] quick-start.md documents the authentication flow completely
- [ ] SKILL.md is under 3000 words
- [ ] No request/response examples appear in SKILL.md (only in references)
